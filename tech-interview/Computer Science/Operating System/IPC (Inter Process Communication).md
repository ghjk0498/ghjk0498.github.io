# IPC (Inter Process Communication)

## 📌 핵심 요약
- **IPC**는 서로 다른 프로세스들이 데이터를 주고받거나 동기화하는 메커니즘
- 운영체제가 제공하는 프로세스 간 통신 방법의 총칭
- 멀티프로세싱 환경에서 프로세스 협업을 가능하게 하는 핵심 기술
- 시스템 자원을 효율적으로 활용하고 복잡한 작업을 분산 처리할 수 있게 함

## 🎯 주요 개념

### 기본 정의
**IPC(Inter Process Communication)**는 독립적으로 실행되는 프로세스들이 서로 정보를 교환하고 협력할 수 있도록 하는 운영체제의 통신 메커니즘입니다.

### 핵심 원리
- **프로세스 격리**: 각 프로세스는 독립된 메모리 공간을 가짐
- **통신 채널**: OS가 제공하는 특별한 통신 경로를 통해 데이터 교환
- **동기화**: 여러 프로세스가 공유 자원에 안전하게 접근하도록 제어

### 주요 특징
- 프로세스 간 데이터 공유 및 통신 지원
- 다양한 통신 방식 제공 (메시지 전달, 공유 메모리 등)
- 동기/비동기 통신 모두 가능
- 운영체제별로 구현 방식이 다를 수 있음

## 📊 상세 내용

### IPC가 필요한 이유

> 💡 **핵심 포인트**: 프로세스는 기본적으로 독립된 메모리 공간을 사용하므로, 직접적인 데이터 공유가 불가능합니다.

1. **보안과 안정성**: 프로세스 격리로 시스템 안정성 확보
2. **협업 필요성**: 복잡한 작업을 여러 프로세스가 나눠서 처리
3. **자원 공유**: 프린터, 파일 등 시스템 자원의 효율적 활용
4. **모듈화**: 큰 시스템을 작은 단위로 분리하여 개발 및 유지보수 용이

### IPC 통신 방식

#### 1. **파이프 (Pipe)**
- 단방향 통신 채널
- 부모-자식 프로세스 간 통신에 주로 사용
- 익명 파이프와 명명된 파이프(FIFO)로 구분

#### 2. **메시지 큐 (Message Queue)**
- FIFO 방식의 메시지 전달 시스템
- 비동기적 통신 지원
- 메시지에 우선순위 부여 가능

#### 3. **공유 메모리 (Shared Memory)**
- 가장 빠른 IPC 방식
- 여러 프로세스가 동일한 메모리 영역 접근
- 동기화 메커니즘 필요 (세마포어, 뮤텍스 등)

#### 4. **소켓 (Socket)**
- 네트워크 통신 기반 IPC
- 같은 시스템 내부 또는 네트워크로 연결된 시스템 간 통신
- TCP/IP, Unix Domain Socket 등

#### 5. **신호 (Signal)**
- 비동기적 이벤트 알림 메커니즘
- 간단한 정보 전달 (신호 번호)
- 프로세스 제어에 주로 사용

#### 6. **세마포어 (Semaphore)**
- 동기화를 위한 카운터 기반 메커니즘
- 공유 자원 접근 제어
- 프로세스 간 실행 순서 조정

### IPC 방식별 비교

| 방식 | 속도 | 복잡도 | 데이터 크기 | 사용 시나리오 |
|------|------|--------|------------|--------------|
| 파이프 | 중간 | 낮음 | 작음-중간 | 단순한 부모-자식 통신 |
| 메시지 큐 | 중간 | 중간 | 작음-중간 | 비동기 메시지 전달 |
| 공유 메모리 | 매우 빠름 | 높음 | 큰 데이터 | 대용량 데이터 공유 |
| 소켓 | 느림 | 중간 | 제한 없음 | 네트워크 통신, 범용성 |
| 신호 | 빠름 | 낮음 | 매우 작음 | 이벤트 알림 |

## 💡 실제 활용

### 적용 사례

#### 웹 서버 아키텍처
- **Apache/Nginx**: 멀티프로세스 모델에서 IPC로 요청 분배
- **프로세스 풀**: 워커 프로세스들이 공유 메모리로 상태 공유

#### 데이터베이스 시스템
- **클라이언트-서버 통신**: 소켓을 통한 SQL 쿼리 전달
- **백그라운드 프로세스**: 공유 메모리로 버퍼 캐시 관리

#### GUI 애플리케이션
- **이벤트 처리**: 메시지 큐로 사용자 입력 처리
- **플러그인 시스템**: 별도 프로세스로 실행되는 플러그인과 통신

### 장단점 분석

**장점**
- 프로세스 격리로 안정성 향상
- 모듈화된 시스템 설계 가능
- 병렬 처리로 성능 향상
- 다양한 프로그래밍 언어 간 통신 가능

**단점**
- 추가적인 오버헤드 발생
- 구현 복잡도 증가
- 동기화 문제 고려 필요
- 디버깅이 어려울 수 있음

### 모범 사례

> 📌 **기억할 점**: IPC 방식 선택은 통신 빈도, 데이터 크기, 지연 시간 요구사항을 고려해야 합니다.

1. **적절한 IPC 방식 선택**
   - 소량 데이터 + 빈번한 통신 → 파이프나 메시지 큐
   - 대용량 데이터 + 빠른 속도 → 공유 메모리
   - 네트워크 확장 가능성 → 소켓

2. **에러 처리 철저히**
   - 통신 실패 시 재시도 메커니즘
   - 타임아웃 설정으로 무한 대기 방지

3. **동기화 주의**
   - 공유 메모리 사용 시 반드시 동기화 메커니즘 구현
   - 데드락 방지 전략 수립
